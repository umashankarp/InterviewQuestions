## The Running Case Study

A tier-1 bank's **Payment Settlement Engine** — a decade-old, on-premises .NET Framework monolith processing SWIFT MT and ISO 20022 settlement instructions between correspondent banks — has grown into a single, tangled codebase where settlement-matching and netting business rules are interleaved directly with ADO.NET `SqlCommand` calls, static utility classes, and inline SWIFT-message-parsing logic. The system works, processes billions of dollars daily, and has never had a settlement-correctness incident — but every regulatory audit takes longer than the last, because auditors cannot trace which specific code enforces which specific control, every new feature risks silently breaking an untested, undocumented business rule, and the last three attempts to add automated test coverage stalled because nothing in the codebase can be tested without a live SQL Server connection. This module walks the refactor of this system into Clean Architecture compliance, end to end.

## Interview Questions

### Basic (8)

1. **Q: Why is a legacy payment-settlement engine a strong worked example for a Clean Architecture capstone, beyond simply being "an example that uses money"?**
   **A:** Because in a regulated financial-services context, the Dependency Rule's core promise — that business logic is isolated, explicit, and independently testable — stops being merely a code-quality nicety and becomes directly load-bearing for regulatory compliance: an auditor asking "show me exactly where and how the four-eyes settlement-approval control is enforced" needs a precise, traceable answer, and a codebase where that control is scattered across a stored procedure, a middle-tier static method, and a UI validation attribute cannot give one credibly — this is the same "declared ≠ actual" theme this entire course has developed, now with a genuine, external, non-negotiable auditor as the party demanding proof, not just an internal engineering standard.
   **Why correct:** States the specific, concrete reason this domain fits a financial-services example better than an arbitrary one (auditability as a non-optional external requirement, not merely a preference), rather than treating "it's about money" as sufficient justification.
   **Common mistakes:** Assuming the fintech framing here is purely stylistic dressing on the same generic Clean Architecture content — the regulatory-audit angle genuinely changes which trade-offs matter most in this specific capstone (Advanced Q5's audit-evidence question, for instance, has no equivalent stakes in a low-stakes internal tool).
   **Follow-ups:** "Would this same argument apply equally to, say, a social-media app's Clean Architecture refactor?" (The Dependency Rule's testability and maintainability benefits, Module 113 Basic Q8, still fully apply — but the specific *external auditor demanding traceable evidence* stake is much sharper and more concrete in a regulated financial context, which is why this module chose it as the capstone's vehicle.)

2. **Q: Describe the legacy system's actual, current structure in concrete terms.**
   **A:** A single ASP.NET Web Forms/MVC hybrid application where a `SettlementController` action method directly opens a `SqlConnection`, executes a stored procedure (`usp_ProcessSettlement`) containing significant embedded business logic (netting calculations, threshold checks) in T-SQL, then calls a static `SwiftMessageParser.Parse(rawMessage)` utility class to extract fields, with validation, netting rules, and SWIFT-field-mapping logic all interleaved in roughly the same few thousand-line files — there is no `Domain` layer, no Repository abstraction, and the only tests that exist are a handful of brittle, slow, full-database-dependent integration tests covering a small fraction of the actual business rules.
   **Why correct:** Gives a concrete, realistic description of the specific anti-patterns present (business logic in stored procedures, static-utility-class parsing, no abstraction layers) rather than a vague "it's messy," setting up the specific, addressable problems the rest of the module works through.
   **Common mistakes:** Assuming a system in this state is unusual or a sign of poor original engineering — this is an extremely common, realistic state for a decade-plus-old, business-critical financial system that has been continuously, safely operating and incrementally patched by many different engineers over many years, precisely why the *approach* to fixing it (Basic Q7) matters so much.
   **Follow-ups:** "Why does business logic ending up in a stored procedure happen so often in exactly this kind of system?" (Stored procedures are often the path of least resistance for a quick performance fix or a rushed feature under production pressure — each individual addition seemed locally reasonable, but the cumulative effect over a decade is business logic completely inaccessible to the application layer's own testing and reasoning.)

3. **Q: What is the first, concrete step in this refactor, and why not simply rewrite the system from scratch?**
   **A:** Apply Module 107's Strangler Fig pattern combined with Branch by Abstraction — introduce an `ISettlementRepository` interface and a `ProcessSettlementUseCase` class *around* the existing legacy logic first, initially with the new interface's implementation still just calling the old stored procedure internally, before any actual business-rule extraction begins; a full rewrite is specifically rejected here because Module 107's own established risk (a live, business-critical financial system processing real, high-value payments daily cannot tolerate the risk of a big-bang cutover to entirely new, unproven logic) applies with unusually high stakes in this domain — a settlement-processing bug isn't merely an inconvenience, it's a real financial-loss and regulatory-reportable-incident risk.
   **Why correct:** Correctly identifies the specific pattern (Branch by Abstraction, introducing the interface before extracting logic) as the actual first step, and gives the domain-specific reason (heightened financial/regulatory stakes) a big-bang rewrite is rejected, directly reapplying Module 107's established risk analysis to this concrete, higher-stakes case.
   **Common mistakes:** Proposing "write the new Clean Architecture version in parallel and cut over once it's done" as the first step — this is exactly the deferred, riskier full-rewrite approach Module 107 already established as the higher-risk option; the correct first step is introducing the abstraction boundary around the *existing, still-running* logic, not building a replacement in isolation.
   **Follow-ups:** "What's the very first, smallest unit of business logic worth extracting this way?" (Basic Q6 — the single highest-value, most audit-scrutinized business rule, so the refactor's earliest, cheapest win directly reduces the team's biggest current pain point rather than an arbitrary starting point.)

4. **Q: Sketch the target ring structure for this specific system, mapping each ring to a concrete piece of the settlement domain.**
   **A:** **Entities:** a `SettlementInstruction` Aggregate (Module 110) encapsulating a settlement's line items, netting state, and approval status; Value Objects for `Money`, `SwiftBIC`, `ValueDate`. **Use Cases:** `ProcessSettlementUseCase`, `ApproveSettlementUseCase`, each orchestrating the Aggregate and Repository. **Interface Adapters:** `SettlementsController` (inbound), `SwiftMessageGateway` translating raw SWIFT/ISO 20022 messages into domain-facing data (an Anti-Corruption Layer, Module 107/109), `EfCoreSettlementRepository`/or a Dapper-based Adapter over the *existing* database. **Frameworks & Drivers:** ASP.NET Core (post-migration from Web Forms), SQL Server, the actual SWIFT network connector.
   **Why correct:** Concretely maps this specific domain's real components onto Module 113's four rings, rather than a generic, unrelated example, demonstrating the ring structure applies naturally even to a decades-old regulated financial system.
   **Common mistakes:** Treating the SWIFT-message-parsing logic as if it belonged in the `Domain` layer because "it's about the business" — parsing a specific wire-format standard is exactly the kind of external-technology concern (Module 113 Basic Q6's ORM-attribute analogy) that belongs in an outer-ring Gateway, translating the external format into the `Domain` layer's own, stable, technology-agnostic shape.
   **Follow-ups:** "Why is `SwiftBIC` modeled as a Value Object rather than a plain string?" (Module 110's tactical-DDD discipline — a BIC has validation rules and equality semantics of its own; wrapping it prevents primitive obsession and gives the `SettlementInstruction` Aggregate a self-validating building block rather than relying on ad hoc string checks scattered elsewhere.)

5. **Q: Why does moving business logic out of a stored procedure and into an Aggregate matter specifically for this system's audit trail, beyond general testability?**
   **A:** A stored procedure's logic is opaque to standard application-level tooling — code review, static analysis, and unit-test coverage tools typically don't reach into T-SQL business logic the same way they cover C# — meaning an auditor asking "prove this netting rule is correctly and consistently enforced" has no artifact beyond reading the raw SQL and trusting it hasn't silently changed; once the same rule lives as a tested method on the `SettlementInstruction` Aggregate, the auditor can be shown a specific, versioned, unit-tested piece of code plus its test suite's pass/fail history as concrete evidence the rule has been continuously, verifiably enforced — not merely present, but demonstrably tested against adversarial and edge-case scenarios (Module 110's adversarial invariant tests).
   **Why correct:** Names the specific, concrete audit-evidence gap (stored-procedure logic being outside normal application tooling's visibility) that moving logic into the Aggregate closes, rather than a generic "testing is good for compliance" claim.
   **Common mistakes:** Assuming a stored procedure with good inline comments is just as auditable as tested application code — comments describe *intent*, not *verified, currently-enforced behavior*; only an actual, currently-passing automated test provides evidence the described behavior is what the system genuinely, presently does, exactly this course's "declared ≠ actual" theme applied to a T-SQL comment specifically.
   **Follow-ups:** "Does this mean stored procedures should never be used in a regulated financial system?" (No — Advanced Q6 covers where stored-procedure-level logic, like a genuinely performance-critical batch calculation, can still be appropriate; the point is that *business-rule* logic specifically needs to live somewhere auditable and testable, which a stored procedure typically isn't, not that SQL itself is forbidden.)

6. **Q: Which specific business rule is chosen as the first piece of logic to extract into the `SettlementInstruction` Aggregate, and why that one first?**
   **A:** The **settlement-netting balance invariant** — "the sum of all matched debit and credit line items within a settlement instruction must net to exactly zero before the instruction can be marked ready for release" — is chosen first because it's simultaneously the single most-frequently audit-scrutinized rule (auditors specifically ask about it every cycle), the rule most engineers already agree is correct but can't currently *demonstrate* is correct without a live database test, and a genuinely synchronous invariant (Module 110's test) squarely belonging on the Aggregate rather than requiring any cross-boundary coordination — making it both the highest-value and lowest-risk first extraction target.
   **Why correct:** Names a specific, concrete business rule and gives the precise, multi-factor justification (highest audit scrutiny, existing but undemonstrable team consensus on correctness, cleanly synchronous per Module 110's test) for choosing it first, rather than an arbitrary starting point.
   **Common mistakes:** Starting the extraction with the most *complex* rule in the system to "get the hard part out of the way first" — Module 107's migration-pattern discipline favors starting with a rule that's valuable, well-understood, and low-risk to extract, building the team's confidence and the fitness-function tooling (Intermediate Q6) before tackling genuinely harder, more ambiguous rules.
   **Follow-ups:** "What would make a candidate business rule a poor choice to extract first?" (One still actively under business dispute or frequent recent change — extracting a rule the team doesn't yet have confident, stable consensus on risks baking an incorrect or soon-to-be-outdated version into the new, harder-to-quickly-patch Aggregate structure.)

7. **Q: Why is a big-bang rewrite specifically riskier for this kind of live, financial, regulated system than for a typical internal business application?**
   **A:** Beyond Module 107's general big-bang-rewrite risks (a long period with no working system, a large batch of unvalidated new logic released all at once), a live settlement engine handling real interbank payments has essentially zero tolerance for a period of incorrect behavior — a bug in a rewritten netting calculation could mean real money moved incorrectly between institutions, triggering not just a customer-facing incident but a regulatory-reportable event, correspondent-bank relationship damage, and potential financial-loss liability; the incremental, Strangler-Fig-based approach (Basic Q3) specifically minimizes the window and blast radius of any single change's potential incorrectness, which matters far more here than in a typical CRUD application where a bug is inconvenient but not existentially costly.
   **Why correct:** Names the specific, elevated stakes (real financial loss, regulatory-reportable incidents, correspondent-relationship damage) distinguishing this system's rewrite risk from a generic application's, giving concrete substance to "it's riskier" rather than asserting it abstractly.
   **Common mistakes:** Treating "this is a critical system" as self-evidently sufficient justification without naming the *specific* categories of harm (financial loss, regulatory reporting, relationship damage) a Principal Engineer would need to articulate concretely when justifying the incremental approach's extra time cost to business stakeholders (Expert Q1).
   **Follow-ups:** "Is there any category of change in this system where a bigger, less incremental step would actually be acceptable?" (Advanced Q4's ATAM-style trade-off analysis covers this explicitly — a change to a clearly-isolated, low-blast-radius component, like an internal reporting dashboard's read-only view, could reasonably tolerate a larger, less incremental change than anything touching the actual settlement-processing/money-movement path.)

8. **Q: What is the very first, concrete, user-and-auditor-visible win the team should expect from even a partial refactor (just the netting invariant extracted)?**
   **A:** A genuine, fast-running, comprehensive unit-test suite for the netting invariant specifically — dozens of edge cases (partial matches, currency mismatches, rounding-boundary scenarios) that were previously untestable without a live database and a slow, brittle integration test now run in milliseconds with zero infrastructure — meaning the next audit cycle's netting-rule question can be answered by pointing directly at a specific, running, currently-passing test suite rather than a walkthrough of T-SQL and verbal assurance, and the next engineer who needs to modify netting logic has immediate, fast feedback rather than needing a full deploy-and-manually-verify cycle.
   **Why correct:** Names a specific, concrete, immediately-realized benefit (a fast, comprehensive test suite for one specific rule) rather than a vague "things get better," tying it back to both the engineering benefit (Module 113 Basic Q8) and the audit benefit (Basic Q5) this capstone has established.
   **Common mistakes:** Expecting the full audit-traceability and architectural benefit to materialize only once the entire system is refactored — Basic Q6's incremental approach means real, usable benefit accrues after each individual extraction, which is itself an important argument (Expert Q1) for why the incremental investment is worth starting even before the full multi-quarter effort completes.
   **Follow-ups:** "How would you demonstrate this specific win to a skeptical stakeholder who doubts the refactor's value early on?" (Show the actual before/after: the old integration test's multi-second, database-dependent run for a handful of scenarios, versus the new unit-test suite's millisecond run covering dramatically more edge cases — a concrete, visible, hard-to-dispute demonstration rather than an abstract architectural argument.)

### Intermediate (8)

1. **Q: Detail the concrete Branch-by-Abstraction plan for introducing `ISettlementRepository` around the existing `usp_ProcessSettlement` stored procedure.**
   **A:** Define `ISettlementRepository` in a new `Domain` project with methods shaped around what the business actually needs (`GetById`, `Save`), *without* yet knowing or caring how they're implemented; create an initial `LegacyStoredProcSettlementRepository : ISettlementRepository` implementation in `Infrastructure` that, internally, still calls `usp_ProcessSettlement` exactly as before — this first step changes *zero* business behavior and carries near-zero risk, but it establishes the seam (Module 107's Branch by Abstraction) through which the actual business logic can later be incrementally moved out of the stored procedure and into a genuine `SettlementInstruction` Aggregate, one rule at a time, with the calling code (`ProcessSettlementUseCase`) never needing to change again after this first step.
   **Why correct:** Gives the precise, sequenced, low-risk first move (introduce the interface and a behavior-preserving legacy-backed implementation before touching any actual logic) that Module 107's Branch by Abstraction specifically prescribes, applied concretely to this system's actual stored procedure.
   **Common mistakes:** Attempting to extract netting logic into the Aggregate in the very same step as introducing the Repository interface — bundling the abstraction-introduction step with the first real logic-extraction step increases the size and risk of that single change; Module 107's discipline specifically separates "introduce the seam" from "migrate the logic through the seam" into distinct, independently-verifiable steps.
   **Follow-ups:** "How would the team verify this first step introduced zero behavior change?" (Run the full existing (if limited) integration test suite against the new `LegacyStoredProcSettlementRepository`-backed path and confirm identical results to the pre-refactor baseline — a direct, if narrow, precursor to Advanced Q3's fuller Parallel Run technique.)

2. **Q: Walk through applying Module 110's synchronous-invariant test to determine the `SettlementInstruction` Aggregate's exact boundary.**
   **A:** "All line items within one settlement instruction must net to zero before release" needs synchronous, same-transaction enforcement — this invariant genuinely spans multiple line items but never spans multiple separate settlement instructions, so all of a single instruction's line items belong *inside* the Aggregate boundary; by contrast, "this settlement instruction's counterparty bank must have an active correspondent relationship" references a separate `CorrespondentBank` concept but doesn't need continuous, synchronous truth for the entire life of the instruction (it's checked at instruction-creation time and, if it later changes, is handled as a separate, explicit business process, not a continuously-re-verified in-Aggregate invariant) — so `CorrespondentBank` is referenced only by ID, exactly as Module 110's own worked `Order`/`Customer` example established.
   **Why correct:** Applies Module 110's specific decision test concretely to this system's own two candidate invariants, correctly distinguishing which genuinely needs to be inside the Aggregate boundary and which doesn't, directly mirroring Module 110's own Order-Customer worked reasoning rather than treating Aggregate-boundary decisions as arbitrary.
   **Common mistakes:** Including the full `CorrespondentBank` relationship status inside the `SettlementInstruction` Aggregate "to be safe" — this recreates Module 110's own Black-Friday incident pattern (an oversized Aggregate spanning unrelated concerns), risking artificial version-conflict contention on high-volume settlement processing exactly as that incident described for order processing.
   **Follow-ups:** "What happens if a correspondent relationship is suspended while a settlement instruction referencing it is still in flight?" (This becomes an explicit, separate business process — likely a Domain Event, Intermediate Q4, reacting to a relationship-suspension event and taking deliberate action on any in-flight instructions referencing that bank — rather than an automatic, implicit consequence of a tightly-coupled Aggregate reference.)

3. **Q: How should the legacy `SwiftMessageParser` static class's tangled parsing-and-validation logic be re-architected as a Gateway/Adapter?**
   **A:** Introduce an `ISwiftMessageGateway` Port in the `Domain`/`Application` ring with a method like `ParseSettlementInstruction(rawMessage) : SettlementInstructionData` (a plain, domain-shaped data structure, not a SWIFT-specific type), and move the existing parsing logic into a `SwiftMessageGateway : ISwiftMessageGateway` Adapter in `Infrastructure` — critically, any *business validation* currently embedded in the same static class (e.g., "reject if settlement amount exceeds a threshold without additional approval") must be separated out and moved into the `SettlementInstruction` Aggregate or `ProcessSettlementUseCase` instead, since that's genuine business logic, not message-format-parsing logic, and belongs in an inner ring regardless of how tangled it currently is with the parsing code it happens to sit next to.
   **Why correct:** Correctly separates two genuinely different concerns tangled together in the legacy static class (format-parsing, a legitimate outer-ring Gateway concern, versus business validation, an inner-ring concern) rather than treating the entire static class as a single, uniform migration unit.
   **Common mistakes:** Moving the entire `SwiftMessageParser` class, business-validation logic included, into the `Infrastructure`-ring Gateway wholesale "since it's all message-handling code" — this would relocate genuine business rules into an outer ring, reproducing Module 114 Advanced Q2's Dependency-Rule-violation risk category in reverse (business logic incorrectly placed outward rather than infrastructure incorrectly placed inward), losing exactly the testability and auditability benefit this entire refactor exists to deliver for that specific validation rule.
   **Follow-ups:** "How would you identify which parts of a tangled legacy utility class are genuinely 'parsing' versus genuinely 'business rules'?" (Ask, for each piece of logic, whether a domain expert (a settlement-operations specialist, not a SWIFT-format specialist) would recognize and care about this specific rule in their own vocabulary — Module 109's Ubiquitous-Language test, reapplied here as a practical legacy-code-triage technique.)

4. **Q: Model the regulatory-reporting requirement — "every completed settlement above a threshold must trigger a regulatory report within 24 hours" — using this domain's established Domain Event pattern.**
   **A:** The `SettlementInstruction` Aggregate raises a `SettlementCompleted` Domain Event (Module 111 Basic Q5) as part of its own `Release()` method's state transition, carrying the settlement's ID, amount, and counterparties; a separate handler — living in the Use Cases/Application ring, triggered via the Outbox pattern (Module 111 Advanced Q2) for reliability — checks the threshold and, if exceeded, initiates the regulatory-report-generation process, entirely decoupled from the `SettlementInstruction` Aggregate's own responsibility, which is only to correctly and completely record that the settlement itself is done; if the reporting requirement's threshold or process later changes, only the handler changes, not the Aggregate's own core settlement logic.
   **Why correct:** Directly reapplies Module 111's Domain Event pattern and Outbox-reliability preview to a genuine, realistic regulatory requirement, correctly keeping the Aggregate's own responsibility narrow (recording settlement completion) separate from the cross-cutting regulatory-reporting reaction, exactly Module 111 Intermediate Q1's Domain-Service/Aggregate-responsibility discipline applied to Domain Events instead.
   **Common mistakes:** Embedding the regulatory-threshold check and report-triggering logic directly inside the `SettlementInstruction.Release()` method itself — this couples the Aggregate's core business concept to a specific, externally-imposed reporting requirement that may change on a completely different schedule (a new regulation, a threshold adjustment) than the underlying settlement-netting business rules ever would, exactly the oversized-Aggregate-responsibility risk Module 110/111 have both already warned against.
   **Follow-ups:** "Why is the Outbox pattern specifically non-negotiable for this particular event, more than for a less regulated notification?" (A regulatory report failing to fire due to a dropped, non-durably-recorded event isn't merely a minor inconvenience — it's a potential compliance violation with real regulatory consequences, making Module 111 Advanced Q2's reliability guarantee genuinely load-bearing here, not just a nice-to-have.)

5. **Q: Design the Parallel Run (Module 107) validating the new Clean-Architecture-based settlement logic against the legacy system before cutover.**
   **A:** For a defined period, every incoming settlement instruction is processed by *both* the legacy stored-procedure path and the new `ProcessSettlementUseCase`/`SettlementInstruction` path in shadow mode — the new path's computed results (netting outcome, approval status, computed regulatory-report triggers) are logged and automatically diffed against the legacy path's actual, production results, with the legacy path's output remaining the only one actually acted upon (payments released, reports filed) during this period; any divergence is immediately flagged for investigation (Advanced Q3's worked example) before the team has any confidence to actually cut real settlement processing over to the new path.
   **Why correct:** Applies Module 107's Parallel Run technique concretely to this system, correctly specifying that the legacy path remains authoritative for real actions during the shadow period and that automated diffing, not manual spot-checking, is the actual verification mechanism — directly reapplying Module 107's own established discipline to a genuinely higher-stakes system where a mismatched cutover could mean real financial harm.
   **Common mistakes:** Cutting over to the new path for a "low-risk subset" of settlements first without full Parallel Run validation, reasoning that a smaller blast radius makes shadow-mode verification less necessary — even a small-volume subset of real settlements carries genuine financial-correctness risk in this domain; the Parallel Run's shadow-mode, zero-real-consequence verification should precede *any* real cutover, regardless of subset size, precisely because of this domain's elevated stakes (Basic Q7).
   **Follow-ups:** "How long should this Parallel Run period last before the team has sufficient confidence to cut over?" (Long enough to observe a representative sample of the system's actual edge-case volume — including month-end/quarter-end high-volume periods and any known rare-but-critical scenarios like partial-netting or multi-currency settlements — rather than a fixed, arbitrary calendar duration; Advanced Q7 covers the final cutover decision criteria in full.)

6. **Q: Design two fitness functions for this system: one enforcing the general Dependency Rule, and one specifically enforcing auditability.**
   **A:** (1) The general Dependency Rule check, directly reusing Module 114 Advanced Q2's `NetArchTest` pattern: `Domain` and `Application` assemblies must have zero dependency on `Infrastructure`, EF Core/Dapper, or ASP.NET Core namespaces. (2) A **compliance-specific** fitness function unique to this domain's stakes: a static-analysis check asserting every public state-transition method on `SettlementInstruction` (`Release()`, `Approve()`, `Reject()`) has a corresponding unit test in the `Domain.Tests` project covering at minimum its primary invariant and at least one rejection/edge case — failing the build if a new state-transition method is added without matching test coverage, converting "every settlement rule is tested" from a point-in-time audit claim into a continuously, mechanically re-verified fact, exactly this course's "declared ≠ actual" theme applied concretely to a compliance assertion rather than a purely architectural one.
   **Why correct:** Supplies both the already-established general fitness function (directly reused, not re-derived, per this domain's non-duplication discipline) and a genuinely new, domain-specific fitness function whose specific assertion (test-coverage-per-state-transition) is calibrated to this system's actual compliance stakes, not just generic code quality.
   **Common mistakes:** Treating a single, generic code-coverage-percentage threshold (e.g., "80% overall coverage") as sufficient compliance evidence — an aggregate coverage percentage says nothing about whether the *specific, individually audit-scrutinized* state transitions are covered; the fitness function needs to assert coverage at the granularity auditors actually care about (per state-transition method), not an average across the whole codebase.
   **Follow-ups:** "What would this compliance fitness function's failure look like in a real pull request, concretely?" (A developer adds a new `PartiallyReverse()` state-transition method to `SettlementInstruction` without a corresponding test in `Domain.Tests` — the fitness-function build step fails immediately, forcing the test to be written before the PR can merge, rather than relying on a reviewer to remember to ask for it.)

7. **Q: What is the specific reconciliation strategy for settlement instructions already in-flight (created under the legacy path) at the moment the new path goes live for new instructions?**
   **A:** Adopt an explicit "no mixed-path settlements" rule: any settlement instruction created before cutover completes its *entire* lifecycle (matching, netting, release, reporting) under the legacy path exclusively, even if that takes days past the cutover date; only *newly created* instructions after cutover use the new Clean-Architecture path — avoiding the significantly higher-risk alternative of migrating an in-flight instruction's partial state mid-lifecycle between two structurally different systems, which risks exactly the kind of subtle, hard-to-detect data-translation bug Module 107's own migration-patterns module (dual-write/CDC reconciliation) already warned about, now with real financial correctness at stake rather than a lower-stakes data migration.
   **Why correct:** Gives a concrete, low-risk reconciliation rule (temporal cutover boundary, no mid-lifecycle instruction migration) specifically calibrated to this domain's stakes, rather than attempting a more complex, riskier live-migration-of-in-flight-state approach.
   **Common mistakes:** Attempting to migrate in-flight settlement instructions' partial state into the new system's `SettlementInstruction` Aggregate representation mid-lifecycle to "complete the cutover faster" — this introduces exactly the translation-correctness risk this rule is specifically designed to avoid, for a marginal, low-value speed gain in a domain where correctness stakes are unusually high.
   **Follow-ups:** "How long might the legacy system need to remain operational under this rule?" (Until every settlement instruction created before cutover has fully completed its lifecycle — likely a period on the order of the system's typical settlement cycle time, directly connecting to Module 107 Advanced Q7's "old systems never die without a deliberate decommissioning plan" finding, here with an explicit, bounded, and business-justified decommissioning timeline rather than an indefinite lingering legacy system.)

8. **Q: As the Principal Engineer leading this refactor, how would you work with the bank's compliance/audit stakeholders throughout the migration, rather than only informing them after the fact?**
   **A:** Involve compliance/audit stakeholders early and iteratively — walk them through the target Aggregate design and the specific compliance fitness function (Intermediate Q6) *before* extraction begins, so they can confirm it will actually produce the evidence artifacts they need at audit time; provide them visibility into the Parallel Run's (Intermediate Q5) divergence-detection results during the validation period, since a clean, well-explained Parallel Run report is itself valuable, concrete audit evidence of the new system's correctness before it ever handles a single real settlement; and explicitly document the reconciliation rule (Intermediate Q7) as a formal, reviewed decision they've signed off on, not an engineering-only choice they discover after the fact.
   **Why correct:** Names concrete, specific engagement points (design walkthrough before extraction, Parallel Run visibility during validation, formal sign-off on the reconciliation rule) rather than a vague "communicate well," treating compliance stakeholders as active design participants whose input shapes the fitness functions and cutover plan, not passive recipients of a finished decision.
   **Common mistakes:** Treating compliance/audit engagement as a final-approval gate at the very end of the refactor, after all technical decisions are already made — this risks a stakeholder objection late enough to be extremely costly to address, when early involvement (as this answer describes) could have surfaced the same concern cheaply, at the design stage.
   **Follow-ups:** "What if compliance stakeholders request a control this refactor's current design doesn't yet support?" (Treat it as a genuine, additional requirement feeding back into the Aggregate/Use-Case design — e.g., a four-eyes-approval requirement might mean `ApproveSettlementUseCase` needs to enforce that the approving user differs from the instruction's creator, a new, explicit invariant added to the design rather than retrofitted awkwardly after the fact.)

### Advanced (7)

1. **Q: Sketch the concrete `SettlementInstruction` Aggregate, `ProcessSettlementUseCase`, `ISettlementRepository`, and the `SettlementCompleted` Domain Event for this system.**
   **A:** **Entities:** `SettlementInstruction` (Aggregate Root) with `Id`, `LineItems: IReadOnlyList<SettlementLineItem>`, `Status` (`Pending`/`Matched`/`Released`/`Reported`), a private pending-events list, and methods `AddLineItem(SettlementLineItem)`, `Match()` (validating the netting-balance invariant, Basic Q6, before allowing the transition), `Release()` (adding a `SettlementCompleted` event); Value Objects `Money`, `SwiftBIC`, `ValueDate`; `ISettlementRepository { GetById, Save }`. **Use Cases:** `ProcessSettlementUseCase` — receives `ProcessSettlementInputData` (raw, Gateway-translated instruction data), constructs/loads the `SettlementInstruction`, calls its domain methods, commits via the Unit of Work (writing state plus the `SettlementCompleted` event to the outbox in one transaction), maps to `ProcessSettlementOutputData`. **Domain Event:** `SettlementCompleted(SettlementId, Amount, Counterparties, CompletedAtUtc)`.

6. **Q: What concurrency and idempotency concerns are specific to payment settlement, and how does the `SettlementInstruction` Aggregate's design address them?**
   **A:** Two, related concerns: (1) **Optimistic concurrency** (Module 110) — if two concurrent processes attempt to transition the same `SettlementInstruction` simultaneously (e.g., a scheduled batch job and a manual operator action both trying to `Release()` it), the Aggregate's version/`RowVersion`-guarded commit ensures only one succeeds, the other retries against the now-current state rather than silently corrupting or double-processing the settlement. (2) **Idempotency** — since SWIFT/ISO 20022 messages can, in rare network-failure scenarios, be redelivered, `ProcessSettlementUseCase` must check an idempotency key (a unique message reference) before creating a new `SettlementInstruction`, returning the existing instruction's result rather than processing the same settlement twice — a concern with no equivalent stakes in Module 110's original `Order`-Aggregate example, since a double-processed settlement instruction means a real, duplicate financial transaction, not merely a duplicated non-financial record.
   **Why correct:** Names both genuinely distinct concerns (concurrency and idempotency) precisely, correctly reapplies Module 110's already-established optimistic-concurrency mechanism, and explicitly identifies idempotency as a new, domain-specific concern this capstone introduces beyond what Module 110's original example needed to address, given the elevated stakes of duplicate financial processing.
   **Common mistakes:** Treating optimistic concurrency control alone as sufficient protection against duplicate processing — concurrency control prevents two *simultaneous* conflicting updates to the *same* Aggregate instance, but does nothing to prevent a *redelivered* message from creating an entirely new, duplicate `SettlementInstruction` in the first place; idempotency-key checking is a separate, additional mechanism specifically needed to close that distinct gap.
   **Follow-ups:** "Where should the idempotency-key check live — the `Domain` Aggregate, the Use Case, or the Gateway?" (The Use Case — `ProcessSettlementUseCase` checks for an existing instruction with the same idempotency key via the Repository *before* constructing a new Aggregate at all, since this is an orchestration-level "have we already handled this request" concern, not an invariant intrinsic to a single `SettlementInstruction`'s own internal consistency.)
